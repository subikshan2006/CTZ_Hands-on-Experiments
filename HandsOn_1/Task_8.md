# Task 8 — Severity vs Priority

**Severity** measures the technical/functional impact of a defect on the system - does it crash
the app, corrupt data, block a core feature?

**Priority** measures how urgently the business wants it fixed, independent of technical
severity.

**Real-world example where High Severity does not mean High Priority:** A rarely-used "Export
to PDF" report generator crashing the whole application (High Severity - a crash) might still be
Low/Medium Priority if that report is used by only one internal team once a quarter and a manual
workaround exists. Conversely, a cosmetic misalignment on the CEO's personal dashboard (Low
Severity - nothing breaks) can be flagged High Priority simply because of who is affected and
organizational visibility.
